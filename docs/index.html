Database Tables Reference

Complete listing of every table in the Tractly schema, its purpose, and its tenant scope.

Table Purpose Scope Key Columns
app_config Global application configuration & metadata (version, maintenance mode, email settings) GLOBAL key, value, type, description
settings Per-tenant configuration (currency, timezone, locale, receipt footer) PER-TENANT tenant_id, key, value
tenants Top-level tenant/business entities GLOBAL id, name, slug, plan, is_active
users Staff user accounts; tenant_id = NULL for superadmins PER-TENANT tenant_id, username, password_hash, role, branch_id
roles Named roles (Admin, Cashier) scoped per tenant PER-TENANT tenant_id, name, is_system_role
permissions Atomic capability definitions (e.g. manage_inventory) GLOBAL name, category, description
role_permissions Many-to-many join: maps roles to permissions per tenant PER-TENANT tenant_id, role_id, permission_id (composite PK)
user_roles Many-to-many join: maps users to roles per tenant PER-TENANT tenant_id, user_id, role_id (composite PK)
branches Physical shop/warehouse locations PER-TENANT tenant_id, name, is_warehouse, is_active
categories Product category groupings with optional default tax rate PER-TENANT tenant_id, name, tax_rate
products Master product catalog with pricing strategies PER-TENANT tenant_id, barcode, cost_price, selling_price, pricing_strategy, target_margin
product_batches Stock batches for FIFO/weighted-average pricing strategies PER-TENANT tenant_id, product_id, batch_number, expiry_date, cost_price
branch_stock Per-branch real-time stock levels PER-TENANT tenant_id, branch_id, product_id, quantity
stock_transfers Inter-branch stock transfer requests with status tracking PER-TENANT tenant_id, from_branch_id, to_branch_id, status
stock_transfer_items Line items within each stock transfer PER-TENANT tenant_id, transfer_id, product_id, quantity
inventory_transactions Full audit log: stock-in, damage, adjustment, sale PER-TENANT tenant_id, product_id, batch_id, type, quantity, reason
invoices Sales invoices with tax, discount, and payment mode PER-TENANT tenant_id, invoice_number, user_id, branch_id, total, payment_mode
invoice_items Individual product lines within each invoice PER-TENANT tenant_id, invoice_id, product_id, quantity, unit_price, line_total
product_meta_fields Custom product attribute field definitions (text, number, date, select) PER-TENANT tenant_id, name, type, is_required
product_meta_values Custom attribute values per product PER-TENANT tenant_id, product_id, field_id, value
product_discounts Time-bound product discounts (percentage or flat) PER-TENANT tenant_id, product_id, discount_type, discount_value, start_date, end_date
taxes Named tax rates applicable to products PER-TENANT tenant_id, name, rate, is_active

Working Model

Tractly operates on a Direct-to-Disk Architecture, combined with a Stateful Reactivity Model in the frontend. This ensures that even with intermittent connectivity, the system remains responsive and data integrity is maintained across all branch nodes.

How to Test This Application

  1. Clone & Install: Clone the repo, navigate to server/, and run npm install.
  2. Start the Backend: Run npm start from server/. On first launch:
    • Database schema is created automatically (all 21 tables).
    • Global app_config keys are seeded: app_version, maintenance_mode, allow_tenant_registration, system_email_from.
    • Per-tenant baseline settings are seeded: currency, timezone, locale, receipt_footer.
    • Default tenant, branch, and admin user are created.
  3. Start the Frontend: Navigate to client/, run npm install, then npm start. Access at http://localhost:4200.
  4. Test Multi-Tenancy: Log in as tech_admin (admin123). Browse products and branches. Log out, log in as mart_admin — you will see a completely different inventory. Verify zero data crossover.
  5. Test RBAC: As tech_admin, go to User Management. Create a new user with the Cashier role. Log in as that cashier and confirm that product management and settings are inaccessible.
  6. Test SuperAdmin: Log in as super (super123). Verify access to the global tenant management panel and system configuration.
  7. Verify Idempotent Seeding: Restart the server a second time. Confirm via logs: "Verified Global Application metadata" and "Verified Baseline Tenant Metadata" appear — but no duplicate rows are inserted (INSERT OR IGNORE is used).
